# 06-高级主题

本页聚焦当前仓库中真正影响企业接入的高级事项：模块、依赖边界、线程安全和已知限制。

## 1. C++23 模块支持

真实文件与 target：

- 模块接口文件：`galay-utils/module/galay.utils.cppm`
- 预导入头：`galay-utils/module/ModulePrelude.hpp`
- 目标名称：`galay-utils-modules`

真实前置条件：

- CMake `>= 3.28`
- `Ninja` 或 `Visual Studio`
- 非 `AppleClang`
- 可用的模块依赖扫描

真实行为：

- 满足条件时才会创建 `galay-utils-modules`
- `BUILD_MODULE_TESTS` 默认 `OFF`
- 显式开启 `BUILD_MODULE_TESTS=ON` 但 CMake `< 3.28` 时，配置会直接失败
- 生成器或编译器不支持时，源码会发出 warning 并关闭模块测试

## 2. umbrella header 与细粒度头文件

### 推荐原则

- 企业项目默认优先细粒度头文件
- 只有在依赖条件满足时再用 umbrella header 或模块

### 原因

- `galay-utils/galay-utils.hpp` 不再默认聚合 `RateLimiter.hpp`
- `RateLimiter.hpp` 仍然无条件包含 `galay-kernel` 与 `concurrentqueue/moodycamel`
- 因此“只想用 String/System”的调用点可以继续使用 umbrella header；只有显式接入限流器时才会带入外部头依赖

## 3. 线程安全与阻塞语义

以下结论基于当前头文件与 `.inl` 实现整理：

| 类型 | 当前实现观察 | 使用建议 |
|---|---|---|
| `ThreadPool` | 内部有互斥锁和条件变量；`waitAll()` 阻塞 | 在普通线程中使用，避免在协程调度线程中阻塞等待 |
| `TaskWaiter` | 通过原子计数与条件变量等待 | `wait()` / `waitFor()` 同样是阻塞调用 |
| `ObjectPool<T>` | 内部加锁；池空时可创建新对象 | 适合非阻塞对象复用 |
| `BlockingObjectPool<T>` | 条件变量阻塞等待 | 仅用于允许阻塞的线程上下文 |
| `CountingSemaphore` / `TokenBucketLimiter` / `SlidingWindowLimiter` / `LeakyBucketLimiter` | 使用原子与等待队列 | 编译仍要求 `galay-kernel` / `concurrentqueue` |
| `RoundRobinLoadBalancer<T>` | `select()` 用原子索引；`append()` 无同步 | 仅在节点集合不再变更时共享 `select()` |
| `WeightRoundRobinLoadBalancer<T>` | 内部权重状态可变且无锁 | 外部同步或单线程使用 |
| `RandomLoadBalancer<T>` / `WeightedRandomLoadBalancer<T>` | 共享 RNG 无锁 | 外部同步或单线程使用 |
| `SignalHandler` | 改写进程级 handler | 适合集中式注册，避免多组件抢占 |

补充：

- 四个 `LoadBalancer::select()` 都返回 `std::optional<T>` 按值，不是零拷贝接口

## 4. 平台与实现边界

- `BackTrace` 只在 Apple/Linux 上有实际栈采集逻辑
- `Process::daemonize()` 主要服务于 POSIX 场景
- `SignalHandler::blockSignal()` / `unblockSignal()` 依赖 POSIX `pthread_sigmask`
- IO 后端由 `cmake/option.cmake` 选择：
  - Linux：默认 `epoll`，显式关闭 `DISABLE_IOURING` 后才尝试 `io_uring`
  - macOS：`kqueue`
  - Windows：`IOCP`

## 5. 已知限制与权衡

### 构建

- `BUILD_MODULE_TESTS` 默认 `OFF`；验证模块导入时再显式开启
- `examples/` 当前提供 `E1-BasicUsage`，满足模块条件时额外提供 `E1-BasicUsageImport`；更深的边界仍主要来自 `test/test_all.cpp`

### 依赖

- 只有 `RateLimiter` 入口不是零外部依赖
- `test_all` 仍要求链接 `galay-kernel`

### 文档与基准

- 当前仓库没有受版本控制的 benchmark 目录；任何旧 benchmark 说法都应视为历史文案
- 历史专题页已并回编号文档；任何历史吞吐量口径都应以本页和 [05-性能测试](05-性能测试.md) 为准

## 6. 企业接入建议

- 在 CI 中分开执行“基线 include 构建”和“模块构建”
- 把 `galay-kernel` / `concurrentqueue` 视为 umbrella / RateLimiter / 模块的明确前置依赖
- 在多线程场景中，只把 `RoundRobinLoadBalancer<T>::select()` 视为“节点集合不变时可共享”；其余共享 balancer 操作都应外部同步
- 以 `test/test_all.cpp` 作为 API 行为回归基线，而不是只信任 Markdown 片段
