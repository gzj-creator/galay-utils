# 01-架构设计

本页描述当前仓库实际导出的入口、目录职责与依赖边界。

## 1. 导出面

| 入口 | 真实文件 / target | 说明 |
|---|---|---|
| 细粒度头文件 | `galay-utils/<module>/*.hpp` | 最稳妥的接入面，依赖边界最清晰 |
| umbrella header | `galay-utils/galay-utils.hpp` | 一次性聚合所有模块，也会连带引入 `RateLimiter.hpp` |
| CMake 接口库 | target `galay-utils` | `INTERFACE` 库，导出 include 目录与 Unix 链接要求 |
| C++23 模块 | `galay-utils/module/galay.utils.cppm` / target `galay-utils-modules` | 仅在工具链满足条件时生成 |

## 2. 目录职责

| 路径 | 角色 |
|---|---|
| `galay-utils/` | 公开头文件与模块接口 |
| `examples/` | 真实可构建示例；当前提供 `E1-BasicUsage`、`E2-SystemProcess`、`E3-ParserBalancer` |
| `benchmark/` | 可复现 micro-benchmark 入口；当前提供 `B1-CoreBench` |
| `test/` | 真实断言与模块烟雾测试 |
| `docs/` | 人类可读文档；当与源码冲突时优先级最低 |
| `cmake/option.cmake` | 平台 IO 后端与构建选项 |

## 3. 依赖边界

### 3.1 CMake target 级别

- `galay-utils` 是 `INTERFACE` 库
- Unix 下自动附带 `pthread`
- Linux 下额外附带 `dl`

### 3.2 头文件级别

- 大多数模块只依赖标准库或平台系统头
- `galay-utils/ratelimiter/RateLimiter.hpp` 无条件包含：
  - `galay-kernel/kernel/Coroutine.h`
  - `galay-kernel/kernel/Waker.h`
  - `galay-kernel/kernel/Timeout.hpp`
  - `galay-kernel/common/Error.h`
  - `concurrentqueue/moodycamel/concurrentqueue.h`
- `galay-utils/galay-utils.hpp` 会包含 `RateLimiter.hpp`
- `galay-utils/module/galay.utils.cppm` 也会导出 `RateLimiter.hpp`

结论：

- “只要用了 umbrella header / 模块，就自动具备额外外部头依赖”
- “只想用轻量模块时，直接包含单个头文件更安全”

## 4. 构建开关

| 选项 | 默认值 | 真实行为 |
|---|---|---|
| `BUILD_EXAMPLES` | `ON` | 构建 `examples/CMakeLists.txt` 中的 target |
| `BUILD_BENCHMARKS` | `OFF` | 开启后构建 `benchmark/CMakeLists.txt` 中的 target |
| `BUILD_TESTS` | `OFF` | 开启后构建 `test_all`，并在满足条件时构建 `test_import_smoke` |
| `BUILD_MODULE_TESTS` | `ON` | CMake `< 3.28` 时会直接导致配置失败；生成器/编译器不支持时会被禁用 |
| `DISABLE_IOURING` | `ON` | Linux 默认走 `epoll`；显式关闭后才尝试 `io_uring` |

## 5. 验证资产

| 资产 | 真实文件 | 作用 |
|---|---|---|
| 示例 | `examples/include/E1-basic_usage.cpp`、`examples/include/E2-system_process.cpp`、`examples/include/E3-parser_balancer.cpp` | 覆盖基础接入、System/Process、Parser/Balancer 三条入口 |
| 基准 | `benchmark/B1-core_benchmark.cpp` | 固定命令可复现的 micro-benchmark 套件 |
| 功能测试 | `test/test_all.cpp` | 大多数模块的行为校验基线 |
| 模块烟雾测试 | `test/module_import_smoke.cpp` | `import galay.utils;` 的最小验证 |

## 6. 平台与特性边界

- `BackTrace` 只在 Apple/Linux 上提供实际栈采集逻辑
- `Process::daemonize()` 仅对 POSIX 语义有意义
- `SignalHandler` 修改的是进程级信号处理器
- 模块构建不支持 `AppleClang`

## 7. 当前权衡

- 仓库同时提供“细粒度头文件”和“全量 umbrella header”，但后者的依赖边界更宽
- `benchmark/` 现提供 `B1-CoreBench` 作为同机回归入口，但它不等同于跨机器 SLA / 吞吐量承诺
- `examples/` 已覆盖三条常见上手路径；更深的边界与异常路径仍主要来自 `test/test_all.cpp`
