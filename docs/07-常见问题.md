# 07-常见问题

## Q1：为什么 README 不再推荐直接运行 `cmake ..`？

因为模块测试是 opt-in 能力，不应该影响 CMake `3.16+` 的基线构建。文档现在统一使用：

```bash
cmake -S . -B build -DBUILD_MODULE_TESTS=OFF
```

## Q2：这个仓库到底是不是“无外部依赖”？

不能笼统地这么说。

- 只包含部分细粒度头文件时，很多模块确实只依赖标准库 / 平台系统头
- 但 `galay-utils/galay_utils.hpp` 会包含 `limiter.hpp`
- `limiter.hpp` 无条件依赖 `galay-kernel` 与 `concurrentqueue/moodycamel`

因此更准确的说法是：**仓库提供很多 header-only 组件，但 `RateLimiter` 入口有额外外部头依赖。**

## Q3：`IniParser` 还存在吗？

不存在。当前公开头 `galay-utils/parser/parser.hpp` 中的真实类型是：

- `ConfigParser`
- `EnvParser`
- `ParserManager`

`.ini` / `.conf` 文件由 `ConfigParser` 负责。

## Q4：真实示例文件和 target 是什么？

- 示例源文件：
  - `examples/include/e1_basic.cpp` → `E1-BasicUsage`
  - `examples/import/e1_basic.cpp` → `E1-BasicUsageImport`（模块条件满足时）
- 功能测试入口：`test/test_all.cpp` → `test_all`
- 模块烟雾测试：`test/import_smoke.cpp` → `test_import_smoke`

其中测试 target 会优先通过 `find_package(galay-kernel CONFIG)` 导入 `galay-kernel::galay-kernel`；在 macOS 默认 `/usr/local` 安装下，`ctest --test-dir build-test` 不需要额外设置 `DYLD_LIBRARY_PATH`。

详情见 [04-示例代码](04-示例代码.md)。

## Q5：什么时候可以用 `import galay.utils;`？

需要同时满足：

- CMake `>= 3.28`
- `Ninja` 或 `Visual Studio`
- 非 `AppleClang`
- 可用的模块扫描工具链

当前模块接口不再默认导出 `limiter.hpp`，因此模块 smoke 不再要求 `galay-kernel` / `concurrentqueue`。如果你业务代码还要用限流器，再单独包含 `galay-utils/ratelimiter/limiter.hpp`。

## Q6：为什么性能页没有吞吐量数字了？

因为当前仓库没有受版本控制的 benchmark 目录或 benchmark target，所以不对外发布性能承诺。详情见 [05-性能测试](05-性能测试.md)。

## Q7：哪些组件需要我自己做线程同步？

基于当前实现，至少包括：

- 任何对 `RoundRobinLoadBalancer<T>` 进行并发 `append()` 的场景
- `WeightRoundRobinLoadBalancer<T>` 的共享实例
- `RandomLoadBalancer<T>` 的共享实例
- `WeightedRandomLoadBalancer<T>` 的共享实例

详情见 [06-高级主题](06-高级主题.md)。

## Q8：哪里可以找到最可信的 API 行为说明？

优先顺序如下：

1. `galay-utils/*.hpp`
2. `examples/include/e1_basic.cpp`
3. `examples/import/e1_basic.cpp`
4. `test/test_all.cpp`
5. `test/import_smoke.cpp`
6. 编号文档页

## Q9：如果我只想用 `StringUtils` / `System`，该怎么避免额外依赖？

如果你只想用轻量模块，可以直接包含具体头文件，例如：

- `galay-utils/string/string.hpp`
- `galay-utils/system/system.hpp`

如果你还需要限流器，再额外包含 `galay-utils/ratelimiter/limiter.hpp`。
